Choosing the Right Parameters for Argon2id Password Hashing

5 minute read Published: 27.6.2026

Argon2id is one of the most secure options for password storage. Its memory-hard design frustrates GPU cracking, yet the algorithm is flexible enough to run efficiently on commodity hardware. Picking good parameters, however, is critical. The goal is to slow down attackers as much as possible while keeping legitimate logins snappy.

Key Parameters in Argon2id

Argon2id exposes three knobs:

  • Time cost (T) – how many times the memory is filled and processed.
  • Memory cost (M) – how much RAM in kibibytes each hash consumes.
  • Parallelism (p) – the number of threads used.

T and M together determine the real cost for an attacker. Raising either one increases the work required to compute a single hash. In practice, you want to push both as high as your environment will allow without degrading the user experience.

Benchmarking Your Environment

Start by measuring how fast Argon2id runs with different values. Most libraries provide a benchmarking mode. With the reference argon2 binary, pass a password on stdin, provide a salt, and select Argon2id explicitly:

echo -n "password" | argon2 somesalt -id -t 3 -m 15 -p 1

This will hash using t=3, m=2^15 KiB (32_768 KiB, or 32 MiB), and p=1. Note the runtime, then adjust -t and -m higher or lower and run again. Record how long each combination takes so you understand the trade-offs.

Aim for a target runtime per hash of somewhere between 200–500 ms on your production hardware. Values in that range are short enough that a user login feels immediate, yet they make large-scale cracking expensive.

Argon2id vs GPU Cracking

Specialized cracking rigs rely on GPUs or FPGAs to compute billions of hashes in parallel. Argon2id's memory hardness undercuts that advantage by forcing each password guess to repeatedly read and write a big region of RAM. GPUs have tremendous compute power but relatively little memory per core. When M is set high, they must constantly swap data, which destroys their throughput. CPUs, with easier access to main memory, handle these workloads with much less slowdown.

Beyond that, Argon2id requires filling memory with pseudorandom data for every password guess. Attackers can't reuse one block of memory for all candidates, so the cost multiplies with each attempt. That property, combined with strong memory requirements, makes large-scale GPU attacks far less attractive.

Balancing T and M

When tuning, prioritize memory first. Increasing M dramatically raises the barrier for GPUs and FPGAs because those platforms are often memory limited. Try to allocate at least 64 MiB if your servers have the RAM to spare. In the reference CLI this is -m 16; in an encoded Argon2 hash, the same setting appears as m=65536 because the PHC string stores memory in kibibytes. Many modern systems can comfortably handle 128 MiB or more for each concurrent hash.

Once memory is set, adjust the time cost. A higher T multiplies the work required without extra memory overhead. For web logins you generally want the combined runtime of T and M to fall under half a second. If you have plenty of CPU headroom you can push toward the upper end of the target range. If your server is under heavy load, dropping T slightly keeps response times consistent. In practice, M shapes the attacker's hardware choices, while T dictates how long each attempt takes. GPUs excel at running thousands of threads but they must share limited memory bandwidth. By forcing every thread to access large chunks of RAM, you neutralize that parallel edge. Once you know how much memory you can spare per hash, increase T until CPU usage or page latency becomes noticeable. This lets you push M as high as possible and then tune T to hit your target runtime without hurting the user experience.

Always test with the same parallelism that production will use. Argon2's m is the total memory for one hash; p splits that memory across lanes or threads rather than multiplying it. Concurrent login requests do multiply memory use, so ten simultaneous 64 MiB hashes can require about 640 MiB before application overhead. Monitor memory usage while benchmarking so you do not accidentally starve other processes.

Example Tuning Session

  1. Start with -m 16 (64 MiB in the reference CLI) and t=2.
  2. Measure the runtime. If it is well below 200 ms, raise m first—try -m 17 (128 MiB in the reference CLI).
  3. If memory is maxed out but runtime is still low, increment t until you approach 300–400 ms per hash.
  4. If the server begins to slow under load, scale back t slightly but keep memory high.

By iterating like this you can zero in on parameters that maximize attacker cost without annoying your users.

Storing Parameters Alongside Hashes

A common practice is to store the chosen t, m, and p with each user's password hash. That way you can increase defaults for new accounts without forcing resets. It also lets you adapt if hardware improves or if you deploy to stronger servers in the future.

Personal Approach

My strategy is simple: begin with OWASP's minimum recommended settings, then push m and t upward until my hardware or response times start to complain. The maximum values the server can handle form the upper bound. From there I dial back toward the minimums until logins feel instantaneous. Meeting in the middle offers a comfortable balance—large enough to frustrate crackers, yet still snappy for real users.

Conclusion

Argon2id's strength comes from its tunable memory and time requirements. For strong security:

  • Allocate as much memory as your servers can tolerate, ideally 64–128 MiB per hash.
  • Increase the time cost until each hash takes roughly 200–500 ms under typical load.
  • Revisit these settings periodically and adjust upward as hardware becomes faster.

A careful balance of T and M provides maximum resistance to password cracking while keeping login performance smooth.
